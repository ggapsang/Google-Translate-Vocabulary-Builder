# 디렉토리 구조 안내
---

## 전체 개요
* `manifest.json`에 정의된 Chrome Extension 구성(권한, 콘텐츠/백그라운드/사이드패널/팝업)을 기반으로 각 폴더가 역할을 분담하고 있음. 기획안 [[01 프로젝트 기획안]]의 “독립 UI 제공”·“학습 도구” 부분과 [[02 기술 스펙]]의 크롬 확장 프로그램 구조를 참조.
* `refs/02 기술 스펙`의 “크롬 확장 프로그램 구조” 섹션에서 제시된 모듈화 전략과 `refs/05 데이터 구조`의 저장/내보내기 스키마가 코드의 각 폴더에 매핑됨.

## 주요 폴더
- `background/`
  * `service-worker.js`는 Manifest의 `background.service_worker`와 연결되어 알림, Spaced Repetition, Google Sheets 연동 준비 등 백그라운드 로직을 처리. 기획안 Phase 2/3 기능과 일치.
- `content/`
  * `content-script.js`와 `styles.css`는 구글 번역 페이지 DOM을 관찰하고 `refs/02 기술 스펙`의 DOM 파싱 전략을 따라 데이터를 추출.
- `sidepanel/` + `popup/`
  * 각각 Manifest의 `side_panel.default_path` 및 `action.default_popup`에 연결된 UI 뷰. `refs/01 프로젝트 기획안`의 “독립 UI”와 “학습 도구”를 구현하고, `refs/03 UIUX 설계`를 구체화.
- `lib/`
  * 재사용 가능한 로직(저장소 래퍼, 파서, CSV/Sheets Export, Spaced Repetition 알고리즘). `refs/02 기술 스펙`의 “데이터 구조” 샘플 객체와 코드 구조도에 대응.
- `assets/`
  * 아이콘 및 정적 자산을 보관. Manifest의 `icons` 및 `action.default_icon`에 참조됨.
- `refs/`
  * 01~05 기획 문서 등 프로젝트의 지식 저장소. 새 문서도 이곳에 두어 코드 변경과 기획을 함께 추적.

## 연결점
* `manifest.json`에서 폴더별 진입점을 확인하고, `web_accessible_resources`로 `lib/html_parsing_mapping_table.json`처럼 콘텐츠 스크립트에 공유되는 자원을 관리.
* 각 기획 문서(`refs/01~05`)의 로드맵·설계·데이터 구조가 해당 폴더와 직접 연결되어 있다는 점을 문서에서 강조하여 “기획안↔코드” 관계를 명확히 밝힘.
